### LOW — 3

**L-1: Crawl batch tasks (SC-75, SC-76) complete quickly but produce zero real product data**  
Crawling merchants that are down, return 403, or have thin catalogs inflates merchant count without filling the product/metadata pipeline. The crawl tasks mark "done" even when crawl quality is 0%. This is known (tiered trust model handles it) but worth watching — if 10+ merchants are Tier C/D, recommendation quality stays poor.

**L-2: schema.d.ts is updated manually when backend runs**  
`npm run gen:api` requires the backend server to be live. In non-running-server contexts, schema updates are hand-edited. This is fragile — a schema field typo would only surface in `tsc -b`, not `npm run build`. The existing pattern (verify with `tsc -b`) mitigates this, but a local OpenAPI JSON snapshot would eliminate the risk.

**L-3: completed_since_reflect resets to 0 but isn't acknowledged by create-tickets**  
Reflection fires at 5 deliveries but the backlog still has 7 ready tasks — no refill needed. The `last_refill_at` field in status.json has never been set (`null`). This is fine but means the refill history is blind. Consider setting it on first run or accepting null as "never refilled."
