## Findings

### 🟡 Important

**F-1 — SC-99 encountered a stale/inaccurate execution plan field reference**
The plan referenced `Merchant.overall_quality_score`, but that field is on `MerchantQualityProfile`. Delivery succeeded because the implementation agent noticed the discrepancy and corrected it. However, inaccurate field references in execution plans increase delivery friction. The architecture has several `via relationship` patterns (quality_profile, shipping_policy) that should be noted in plans as join requirements.

**F-2 — SC-101 surfaced two foot-guns in the merchant import CLI**
`szimpla add-merchant` without `--crawl-now` does not crawl, and `crawl_merchant()` does not commit. Both were only discovered during delivery. These are well-understood sharp edges now (documented in SC-101 memory), but they're not surfaced in the CLI help text or in the merchant admin workflow. A future ticket could add `szimpla add-and-crawl` or make the default behavior explicit.

**F-3 — Batdorf & Bronson (custom platform) contributed 0 products despite successful import**
Custom-platform merchants are a known dead zone for the current adapter set. Tier C assignment is correct. If this pattern recurs in the next merchant expansion batch, a short-circuit that skips import entirely for merchants with unknown/custom platforms would avoid polluting the active merchant count.

**F-4 — Non-coffee product tagging (SC-102) dramatically improved the fill-rate denominator**
155 non-coffee products were tagged and removed from the normalization denominator. This alone moved roast_unknown from 259→175 and process_unknown from 451→233 on all active products without any new extraction logic. **The denominator fix matters as much as new patterns.** Future parser improvement tickets should audit the non-coffee boundary first.

**F-5 — Origin coverage ceiling persists at ~35% for coffee-only products**
This is a data availability problem, not a parser problem. Product descriptions are not stored, so the only signal is the product title. Unless descriptions are persisted (requires schema change + re-crawl pass), this ceiling is structural. The Phase 2 goal `≥70% origin fill` was set before this ceiling was understood. It should be revised.
