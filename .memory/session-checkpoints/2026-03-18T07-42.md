## Root Cause Diagnosis

### The Engine Was Never Broken
The pipeline was producing results all along. The issue: `variant_deal_facts` table shows 0 rows
in the DB when inspected outside of a live API request. This led to the false assumption that the
engine would fail at the "no fact" step.

### How It Actually Works
`materialize_variant_deal_facts()` is called at the start of every `build_recommendations()` call.
It materializes facts into the SQLAlchemy session via `session.flush()`. Since `build_recommendations`
runs within the same session, the in-memory flushed facts are available via the session cache when
the `fact_by_variant.get(variant.id)` lookup happens. The facts persist in the DB only after
`session.commit()` is called (which the API endpoint does).

This means:
- Via API: facts are committed to DB after each recommendation run ✅
- Via direct DB inspection between runs: table shows 0 (or stale) rows ❌ (misleading)
- The SC-68 fix (wiring `materialize_variant_deal_facts` to post-crawl hook) is still correct and
  ensures facts are pre-populated for the biggest-sales query too.
