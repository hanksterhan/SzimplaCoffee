# SC-68 Delivery Checkpoint — 2026-03-18T07:24 PDT

## Ticket
SC-68: Ensure VariantDealFact rows are created after each crawl

## What Changed
**File:** `backend/src/szimplacoffee/services/crawlers.py`

Added 8 lines to the post-crawl success block in `crawl_merchant()`, immediately after the existing SC-32 quality scorer hook:

```python
# SC-68: materialize deal facts so recommendation engine has rows to join on
try:
    from .recommendations import materialize_variant_deal_facts as _materialize_deal_facts

    _materialize_deal_facts(session)
except Exception:
    pass  # deal fact failure must not break the crawl result
```

## What Was Found (Audit)

- `variant_deal_facts` table existed in the DB but had **0 rows** before this fix
- `materialize_variant_deal_facts()` in `recommendations.py` is a full implementation that:
  - Iterates all active product variants
  - Calls `_materialize_variant_deal_fact()` per variant (upserts VariantDealFact rows)
  - Calls `session.flush()` to persist
- This function was **only called inside `run_recommendation()`** — never after crawls
- As a result, the deal facts table was only populated when someone ran a recommendation, not after data was freshly crawled

## Root Cause
Deal fact computation was decoupled from the crawl pipeline. The recommendation engine would call `materialize_variant_deal_facts()` inline during a recommendation run (lines 552, 621 of recommendations.py), but the table would remain stale between runs. If no recommendation run had ever happened, the table was empty.

## Fix Pattern
Followed the same lazy-import + try/except pattern established by SC-32 (quality scorer hook). This ensures:
- Deal fact failure cannot break crawl results
- No import cycle at module load time
- Minimal diff (8 lines)
