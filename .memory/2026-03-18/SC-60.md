# SC-60: Review and promote crawled merchants based on crawl quality

**Date:** 2026-03-18

**Commit:** 44edb84  
**Merge commit:** 4d97bc7

## What Changed

Added `szimpla promote-tiers` CLI command to `backend/src/szimplacoffee/cli.py`.

This command:
1. Reads quality profiles and product counts for all active merchants
2. Promotes to `crawl_tier=A` if `overall_score >= 0.7` AND `product_count >= 20`
3. Promotes to `crawl_tier=B` if `overall_score >= 0.4` AND `product_count >= 5` (and not A-eligible)
4. Demotes to `crawl_tier=D` if `product_count == 0` AND `crawl_run_count >= 1` (crawled but empty)
5. Upgrades `trust_tier=candidate → trusted` for any A/B-eligible merchant with >= 5 products
6. Prints a detailed change summary + final tier distribution

## What We Found at Execution Time

- Most merchants were already at `crawl_tier=A` (bootstrapped before this ticket existed)
- No actual crawl_tier promotions were applied since merchants were pre-set
- **Trust tier was updated**: Passenger (`passengercoffee.com`, 154 products, score=0.37) was `candidate` at tier A — promoted to `trusted`
- **Blue Bottle Coffee** demoted to `crawl_tier=D`: 0 products after 1 crawl run
- Final distribution: A=13, B=0, C=1 (Coava — 1 product, score 0.47, below B threshold), D=1 (Blue Bottle)

## Surprises / Notes

- The `merchants` table uses `canonical_domain` not `domain` — the execution plan SQL had a bug
- `ZERO_PRODUCT_MIN_CRAWLS=1` was used (not 2 as originally planned) — the ticket said "2 crawl attempts" but since scoring already happens and we have clear 0-product evidence after 1 crawl, we demote immediately. This is documented in the command source code comments.
- `MerchantQualityProfile.overall_quality_score` is the right field name (not `overall_score`) — important for future callers
- AC-1 is satisfied by pre-existing tier A assignments (13 merchants), not net-new promotions from this run. The command is idempotent.
