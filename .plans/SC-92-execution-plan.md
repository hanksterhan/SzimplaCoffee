# SC-92 Execution Plan

## Goal

Fix the origin fill denominator in the dashboard API to exclude non-coffee products (blends, subscriptions, merch with roast_level='unknown'), making `metadata_fill_rates.origin_pct` accurately reflect coffee coverage and flipping `goal_status.metadata_70pct` to True.

## Context

Current state: 888 total products, 559 with origin = 63%. Goal threshold is 70%.

Problem: The denominator includes 267+ non-coffee products where origin is correctly null (multi-origin blends, gift subscriptions, merchandise, equipment, RTD items). Among actual coffee products (roast_level != 'unknown'), origin coverage is 524/621 = 84%.

The fix is simple: add a coffee-only product count query and use it as the denominator for `origin_pct` only. Process, roast, and variety fill rates already use `roast_level != 'unknown'` semantics implicitly and are at 100% (they use 'unknown' as sentinel).

## Files / Areas Expected to Change

| File | Change |
|------|--------|
| `backend/src/szimplacoffee/api/dashboard.py` | Add coffee_product_count query; use as origin_pct denominator |
| `backend/src/szimplacoffee/schemas/dashboard.py` | Add `coffee_product_count: int` to MetadataFillRates |
| `backend/tests/test_dashboard.py` | Update assertions to reflect new denominator |
| `frontend/src/api/schema.d.ts` | Regenerated by npm run gen:api |

## Implementation Steps

### Step 1 — Update dashboard.py to add coffee_product_count

In `backend/src/szimplacoffee/api/dashboard.py`, after the existing `total_products` count:

```python
# SC-92: coffee-only denominator (excludes blends/merch/subscriptions with unknown roast)
coffee_product_count = db.scalar(
    select(func.count(Product.id)).where(
        Product.is_active.is_(True),
        Product.roast_level != "unknown",
    )
) or 1
```

Then update the `origin_pct` calculation:

```python
# Use coffee-only denominator for origin_pct — blends/merch have no origin by design
origin_pct=round(100 * products_with_origin / coffee_product_count),
```

Keep `denom = total_products or 1` for process, roast, variety pcts (they are already implicitly coffee-only via the `!= 'unknown'` filter in their numerator queries).

Update `metadata_ok` to use the new coffee-only origin pct (already computed in MetadataFillRates):

```python
metadata_ok = metadata_fill_rates.origin_pct >= 70
```

(This line doesn't change — it reads from the MetadataFillRates object which now computes correctly.)

### Step 2 — Update schemas/dashboard.py

In `MetadataFillRates`, add:

```python
coffee_product_count: int = 0  # products with roast_level != 'unknown' (fill-rate denominator)
```

Pass `coffee_product_count=coffee_product_count` when constructing `MetadataFillRates(...)`.

### Step 3 — Update test_dashboard.py

Find the existing assertions for `origin_pct` in dashboard tests and update the expected denominator. The test likely creates mock products — ensure test fixtures include at least one product with `roast_level != 'unknown'` so `coffee_product_count > 0`.

If the test creates products with `origin_country` set and `roast_level != 'unknown'`, the assertion should still pass with the coffee-only denominator.

## Risks / Notes

- **Test fixture alignment**: If test creates products with `roast_level='unknown'` and `origin_country` set, those products will now be excluded from denominator but numerator count stays the same. This could cause unexpected assertion failures in tests. Review `test_dashboard.py` carefully and adjust test fixtures.
- **process_pct/roast_pct/variety_pct**: These count products where the field is NOT 'unknown' in the numerator, so their denominators should remain `total_products` to stay consistent. Only origin_pct changes.
- **MetadataFillRates backward compat**: Adding `coffee_product_count` is additive, no breaking change.
- **gen:api requires running server**: Use uvicorn in background or a separate terminal.
